One Claude Code skill that unifies four open-source architecture standards — C4 + Structurizr, ArchiMate, TOGAF ADM, and arc42 + ADRs — to diagram, document, review, and model architecture as code.

View on GitHub → See live examples

Why it exists

The open-source ecosystem has good single-purpose skills — C4+ArchiMate plugins, arc42 toolkits, ADR generators. But none unify all four frameworks, and none cover TOGAF or architecture review. This skill fills that gap and keeps one traceable model under every diagram, doc, and decision.

C4 + Structurizr

How is this system built?

Context · Container · Component diagrams as code.

ArchiMate 3.x

How does the enterprise fit?

Capabilities → applications → technology, across systems.

TOGAF ADM

How do we deliver change?

10-phase method, roadmaps, portfolio (TIME).

arc42 + ADR/MADR

How do we write it down?

System docs and decision records, docs-as-code.

Four modes

The skill routes any architectural request to the right framework at the right altitude.

Mode 1

Diagram

C4 / Structurizr DSL / Mermaid / PlantUML — Context → Container → Component, with diagram hygiene baked in.

Mode 2

Document

arc42 sections (LEAN / ESSENTIAL / THOROUGH) and ADR/MADR decision records.

Mode 3

Review / assess

ISO/IEC 25010 quality attributes + EA principles → severity-graded findings + a verdict.

Mode 4

Model the enterprise

Capability → app → technology with realization links; TOGAF ADM structure & portfolio rationalization.

A stable URN-style ID scheme — ea:org:system:kind:name — keeps one model under everything, plus a scripts/ea_lint.py cross-artifact consistency checker.

Example 2 — Architecture Decision Record

Mode 2 · Document · MADR

“We need to pick a messaging system for our order events — Kafka, RabbitMQ, or AWS SNS/SQS. The team has no Kafka ops experience but we expect ~10k events/sec and want event replay. Write up the decision.”
✔ MADR structure + status✔ decision drivers✔ all 3 options w/ pros & cons✔ justified outcome✔ trade-offs recorded

The skill recognised this as one architecturally-significant decision and produced a proper MADR — title, status, decision drivers, three options with pros/cons, a justified outcome, and consequences including the trade-offs accepted:

# 0001. Use AWS SNS/SQS for order events

- Status: proposed
- Date: 2026-06-18
- Tags: messaging, order-events, eventing

## Decision Drivers
- Throughput: ~10k events/sec with headroom
- Event replay: re-read history from an arbitrary point
- Operational burden: team has no Kafka ops experience
- Time-to-value; ordering & delivery semantics; cost; reversibility

## Considered Options
1. Apache Kafka (self-managed / MSK / Confluent)
2. RabbitMQ (with streams plugin)
3. AWS SNS + SQS (+ Kinesis / EventBridge Archive for replay)

## Decision Outcome
Chosen: "AWS SNS/SQS with a managed log for replay where needed" —
best fit for a team with no Kafka ops while still meeting throughput
and giving a credible replay story without operating brokers.

### Consequences
- Good: minimal ops; elastic & managed; fast time-to-value; pay-per-use
- Bad: replay isn't uniform (needs Kinesis layer); AWS lock-in; FIFO quotas to verify
- Neutral: requires idempotent consumers & an explicit ordering key (orderId)

Example 3 — Architecture review

Mode 3 · Review / assess

“Review our architecture: a single Django monolith on one EC2 instance, one PostgreSQL with no replica, secrets in committed settings.py, no caching, a synchronous third-party email call in the request path, no logging/metrics, ~50k DAU. Is this design sound?”
✔ explicit verdict✔ severity-graded✔ evidence + fix per finding✔ balanced + prioritized

Verdict: Needs revision — the monolith is the right altitude for 50k DAU; the problems are security & operational hygiene, not the decomposition. Findings, abbreviated:

  • 🔴 Critical — committed secrets in settings.py → rotate now, move to Secrets Manager/SSM, purge history, add a pre-commit scanner.
  • 🔴 Critical — single EC2 instance = SPOF → ≥2 instances across ≥2 AZs behind an ALB in an ASG.
  • 🔴 Critical — single un-replicated Postgres → RDS/Aurora Multi-AZ + tested PITR backups.
  • 🟠 Major — no observability → structured logs + RED metrics + Sentry + one alert.
  • 🟠 Major — synchronous email in request path → move to a Celery/SQS queue with retries & timeouts.
  • 🟠 Major — no caching & no stated SLOs → ElastiCache for hot reads; write down availability/latency/RPO/RTO.
  • 🔵 Suggestion — stay a monolith; record the infra decisions as ADRs (Mode 2); draw the target-state C4 diagram (Mode 1).
